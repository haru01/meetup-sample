# Claude Code クイックスタート — MTP-010 イベント作成で学ぶ AI 駆動開発

プログラミング経験がある方が Claude Code を使って、**ストーリー1つをフルスタック（API・UI・E2E）で実装する流れ**を体験するチュートリアルです。

題材: [MTP-010 イベント作成](../docs/stories/meetup-todo.md#mtp-010-イベント作成)（実装済み — 既存コードを参考に学習できます）

所要時間: 約 60 分

> **Note**: MTP-010 は既にリポジトリに実装済みです。このチュートリアルは AI 駆動開発のワークフローを学ぶためのガイドとして活用してください。自分で手を動かしたい場合は、MTP-011（イベント公開）以降の未実装ストーリーで同じ手順を試すことをおすすめします。

---

## 前提

- Claude Code がインストール済み（[公式ガイド](https://code.claude.com/docs/en/getting-started)）
- このリポジトリを clone 済み
- Docker Desktop が起動済み

```bash
git clone <repository-url>
cd meetup-sample
./scripts/docker-dev.sh up
./scripts/docker-dev.sh install
```

---

## 1. まずコードベースを理解する（5分）

実装を始める前に、Claude Code に既存コードの構造を聞いてみましょう。

### やること

Claude Code を起動し、以下のような質問をする:

```
claude
```

```
このリポジトリの構成を教えて。backend の community コンテキストはどういうレイヤー構造になっている？
```

### コツ

- **Plan Mode（Shift+Tab）** で聞くと、コードを変更せず安全に調査できる
- `@docs/stories/meetup-todo.md` のようにファイルを指定すると、ピンポイントでコンテキストを渡せる
- 「シニアエンジニアに聞くように質問する」のが公式推奨

### 確認ポイント

- `controllers/` → `usecases/` → `models/` のレイヤー構造を把握できたか
- `Result<T, E>` パターン、Branded Types、Zod スキーマの存在を確認できたか
- フロントエンドの `pages/` / `components/` / `hooks/` の構成を理解できたか

---

## 2. AI と計画を壁打ちする（15分）

いきなりコードを書かせるのではなく、まず「何をどう作るか」を AI と一緒に考えます。

### やること

MTP-010 のストーリーを渡して、実装計画を相談する:

```
@docs/stories/meetup-todo.md の MTP-010 イベント作成を実装したい。
既存の MTP-005 コミュニティ作成のパターンを参考に、実装計画を立てて。
backend → frontend → e2e の順で。
```

### コツ

- **What（何を作るか）+ Why（なぜそう作るか）+ 既存パターンへの参照**を伝えるのが効果的
- 「MTP-005 を参考に」と言うだけで、AI は既存コードを読んでパターンを踏襲する
- 計画が出たら **Ctrl+G でエディタで開いて編集できる** — 不要なステップを削ったり順番を入れ替えたりしてから承認する

### 計画に含まれるべきもの

| レイヤー | 作成するもの |
|---------|------------|
| Backend モデル | `event.schema.ts`（Zod スキーマ）、`event.ts`（型・ファクトリ関数） |
| Backend UseCase | `create-event.usecase.ts` |
| Backend Controller | `event-controller.ts`、`event-openapi.ts` |
| Backend Repository | `event-repository.ts` |
| Prisma | `Event` モデルをスキーマに追加 |
| Frontend | `EventCreateForm.tsx`、`EventCreatePage.tsx`、ルーティング追加 |
| E2E | `event.spec.ts` |

### 壁打ちで聞くと良い質問

計画が出たあと、以下のような質問で深掘りすると計画の精度が上がる:

- 「イベントの開催形式（ONLINE / OFFLINE / HYBRID）はどこで定義する？コミュニティの Category と同じパターン？」
- 「Prisma スキーマの Event モデルと Community モデルのリレーションはどうなる？」
- 「DRAFT 状態のイベントは一覧 API で返す？返さない？MTP-015 の受け入れ条件と整合する？」

> **なぜ計画を壁打ちするのか？**
> 計画なしでいきなり「実装して」と頼むと、AI は全体の整合性を考えずにファイルを作り始めることがある。5分の壁打ちで30分の手戻りを防げる。

## 3. AI にコードを書かせる（25分）

計画を承認したら、実装に移ります。一度に全部頼まず、レイヤーごとに進めるのがポイントです。

### 3-1. Backend: モデル + UseCase（10分）

```
MTP-010 の backend を実装して。まず models/ と usecases/ から。
既存の community コンテキストと同じパターンで作って。
```

#### コツ

- **小さく頼んで、確認してから次へ**。「backend 全部作って」より「モデルと UseCase から」の方が軌道修正しやすい
- 生成されたコードの diff を確認する。特に:
  - `Result<T, E>` を返しているか（例外を throw していないか）
  - UseCase にドメインロジックが入り込んでいないか（オーケストレーションのみか）
  - Zod スキーマから型を導出しているか（`z.infer<typeof ...>`）

#### テストを実行して確認

```
backend のテストを実行して
```

AI が生成したテストが通ることを確認する。失敗したら、エラーメッセージを見てどこが問題か AI と一緒に修正する。

### 3-2. Backend: Controller + OpenAPI（5分）

```
次に controller と openapi 定義を作って。express-openapi-validator で検証されるようにして。
```

### 3-3. Frontend: フォーム + ページ（7分）

```
次に frontend のイベント作成フォームとページを作って。
CommunityCreateForm.tsx と CommunityCreatePage.tsx を参考にして。
```

#### コツ

- フロントエンドは **参考ファイルを具体的に指名** すると、スタイルや構造が揃いやすい
- 「Tailwind のクラスは既存コンポーネントと揃えて」と伝えるのも有効

### 3-4. E2E テスト（3分）

```
最後に e2e テストを作って。community.spec.ts のパターンに合わせて。
オーナーがイベント作成フォームを入力して作成成功するフローをテストして。
```

### 途中で期待と違う出力が出たら

1. **Esc で即座に止める** — AI が間違った方向に進んでいるのを見たら、完了を待たずに止める
2. **何が違うか具体的に伝える** — 「違う」ではなく「UseCase にバリデーションロジックが入っている、models/ に移して」
3. **2回修正しても直らなければ `/clear` して再出発** — コンテキストが汚れている可能性がある

---

## 4. AI とコードレビューする（15分）

実装が一通りできたら、コミット前にレビューします。

### 4-1. diff を確認する

```
今回の変更の diff を見せて
```

diff を読んで、以下を自分の目で確認する:

- [ ] ファイルの配置は計画通りか
- [ ] 不要なファイルが生成されていないか
- [ ] import パスは既存のエイリアス（`@shared/`、`@auth/` 等）を使っているか

### 4-2. AI にレビューを依頼する

```
今回の変更をレビューして。以下の観点でチェックして:
- AGENTS.md の Core Patterns に準拠しているか
- Result 型を正しく使っているか
- UseCase がオーケストレーションのみか（ドメインロジックが混入していないか）
- OpenAPI 定義でドメイン制約を重複定義していないか
```

#### コツ

- **レビュー観点を具体的に指定する** — 「レビューして」だけだと表面的な指摘しか出ない
- AGENTS.md や CLAUDE.md のルールを引用すると、プロジェクト固有の基準でレビューできる
- 指摘が出たらその場で修正を依頼する:「指摘の 1, 3 を修正して」

### 4-3. テストで最終確認

```
backend のテストと e2e テストを実行して、すべて通ることを確認して
```

### 4-4. コミットする

すべて通ったら:

```
/commit
```

> **なぜ AI にレビューを頼むのか？**
> 自分で diff を読むのは前提。その上で AI にレビューを頼むと、見落としがちなパターン違反やエッジケースを拾える。AI は「動くか」だけでなく「プロジェクトの規約に合っているか」もチェックできる。

## 振り返り

このチュートリアルで体験した AI 駆動開発のサイクル:

```
理解 → 計画 → 実装 → レビュー → コミット
```

### うまくいくコツまとめ

| フェーズ | コツ |
|---------|------|
| 理解 | Plan Mode で安全に調査。シニアエンジニアに聞くように質問する |
| 計画 | What + Why + 既存パターン参照で伝える。計画を Ctrl+G で編集してから承認 |
| 実装 | レイヤーごとに小さく頼む。diff を確認してから次へ進む |
| レビュー | 観点を具体的に指定する。AGENTS.md のルールを引用する |
| 軌道修正 | Esc で止める。2回修正して直らなければ `/clear` |

### よくある失敗パターン

| パターン | 対策 |
|---------|------|
| 「全部作って」と丸投げ | レイヤーごとに分割して依頼する |
| diff を見ずに次へ進む | 各ステップで変更内容を確認する習慣をつける |
| エラーが出たら即 AI に丸投げ | まず自分でエラーメッセージを読み、仮説を持ってから聞く |
| 1セッションに詰め込みすぎ | タスク間で `/clear` してコンテキストをリセットする |
